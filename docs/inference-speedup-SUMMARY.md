- In sbatch wrappers, capture python's exit (`RC=$?; ... ; exit $RC`) — a trailing `echo` makes SLURM report COMPLETED 0:0 even when python failed (masks errors).
- Don't run a SLURM **array** over DIFFERENT configs/datasets — `parallel_proteina_inference` auto-detects the array as shards of ONE run → deadlocks "Waiting for N inference step shards (0/N done)" forever. Use SEPARATE jobs + `--num_shards 1 --shard_index 0 --no-dynamic_resharding`. (Array is ONLY for sharding a single dataset, like prod wc_badafdb.) On a non-SLURM host (A6000), `--num_shards N --shard_index {0..N-1}` as N SEPARATE parallel process launches (one per GPU) IS the correct way to shard one dataset+config across GPUs — `shard_proteins()` LPT-bin-packs by `const + c*L²*log(L)` cost, not naive round-robin.
- ⛔⛔ **Never retry a background launch (`nohup ... &`) that looks like it failed without first verifying via `ps`/`nvidia-smi` whether it actually started.** An error printed AFTER the `&` (e.g. a `mkdir`/pidfile race) does not mean the backgrounded process didn't launch — it can have already detached successfully. A blind retry launches a SECOND full copy that races the first on shared state (lost ~11h of GPU, 2026-07-02). **Fix, now standing practice:** any autonomous driver script launched via `nohup` must acquire a PID lock file (`kill -0` liveness check + `trap ... EXIT` cleanup) at its own top — makes duplicate launches structurally impossible regardless of how the launch attempt is invoked or misread.
- If a driver DOES get duplicated: killing the leaf GPU processes is NOT enough — once a bash-wrapper parent dies, its python children get reparented to PID 1 and keep running/spawning. Enumerate the FULL process tree (`ps -eo pid,ppid,cmd`, follow every ppid chain) before considering cleanup done. Only kill PIDs with verified, unambiguous command-line evidence (exact matching args/paths) — never a broad `pkill -f` pattern on a shared host.
- Trust (but verify once, don't assume) that `run_prediction_pipeline.py`'s resume logic is SAMPLE-COUNT-aware, not just presence-aware: `find_proteins_needing_proteinebm`/`_proteinebm_csv_text_complete` compares the analysis CSV's row count against the CURRENT tar member count, so a clean relaunch after a partial/interrupted run correctly re-runs stale analysis (e.g. after a smoke test's small N gets topped up to the real N) — no manual deletion needed.
- The tar-corruption guard in `protein_tar_utils.py` ("refusing to re-tar, would drop N members already present") is a real backstop that has prevented actual data loss during a concurrent-write race — but it's a backstop, not a design pattern; still avoid concurrent access to the same protein dirs on purpose.
- Don't call `score_structure()` directly on raw CA-only PDBs in a naive loop (e.g. for benchmarking) — it hits an expensive PER-CALL cg2all reconstruction fallback (7-36s/call) meant only for direct callers. Pre-reconstruct once via `reconstruct_all_atom()`/`_reconstruct_ca_only_pdbs()` OUTSIDE the timed loop, pass `_allatom_pdb=<path>` into every call — matches the real batch pipeline's design (`score_proteina_structures_openfold`). Same bug independently existed in the ColabDesign-side scorer (`af2rank_scorer.py`, a separate implementation, doesn't share code with the OpenFold-side scorer).
- **AF2Rank precision on A6000+ (bf16-capable hardware): use `precision="bf16"` EAGER, not compiled, not fp16.** fp16 is NOT supported by OpenFold's custom `attention_core` CUDA kernel (`SUPPORTED_DTYPES = [float32, bfloat16]` — hard kernel limitation, raises immediately). Combining `compile_inference_path=True` with `precision="bf16"` makes it SLOWER than bf16 eager, not faster (per-block compile's Python/dynamo-guard overhead dominates once bf16 tensor cores already make ops fast) — don't bother compiling bf16.
- After the length-bucketing feature (bucket=32) shipped: any real-workload compile-vs-eager AF2Rank benchmark is comparing padded-compiled against unpadded-eager (padding only applies to the compiled side) — NOT apples-to-apples with pre-bucketing benchmark numbers. A length landing just under a bucket ceiling pays a real, sometimes-large compute tax on the compiled side alone.
