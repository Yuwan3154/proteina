- ‚õî‚õî **Never retry a background launch (`nohup ... &`) that looks like it failed without first verifying via `ps`/`nvidia-smi` whether it actually started.** An error printed AFTER the `&` (e.g. a `mkdir`/pidfile race) does not mean the backgrounded process didn't launch ‚Äî it can have already detached successfully. A blind retry launches a SECOND full copy that races the first on shared state (lost ~11h of GPU, 2026-07-02). **Fix, now standing practice:** any autonomous driver script launched via `nohup` must acquire a PID lock file (`kill -0` liveness check + `trap ... EXIT` cleanup) at its own top ‚Äî makes duplicate launches structurally impossible regardless of how the launch attempt is invoked or misread.
- If a driver DOES get duplicated: killing the leaf GPU processes is NOT enough ‚Äî once a bash-wrapper parent dies, its python children get reparented to PID 1 and keep running/spawning. Enumerate the FULL process tree (`ps -eo pid,ppid,cmd`, follow every ppid chain) before considering cleanup done. Only kill PIDs with verified, unambiguous command-line evidence (exact matching args/paths) ‚Äî never a broad `pkill -f` pattern on a shared host.
- Trust (but verify once, don't assume) that `run_prediction_pipeline.py`'s resume logic is SAMPLE-COUNT-aware, not just presence-aware: `find_proteins_needing_proteinebm`/`_proteinebm_csv_text_complete` compares the analysis CSV's row count against the CURRENT tar member count, so a clean relaunch after a partial/interrupted run correctly re-runs stale analysis (e.g. after a smoke test's small N gets topped up to the real N) ‚Äî no manual deletion needed.
- The tar-corruption guard in `protein_tar_utils.py` ("refusing to re-tar, would drop N members already present") is a real backstop that has prevented actual data loss during a concurrent-write race ‚Äî but it's a backstop, not a design pattern; still avoid concurrent access to the same protein dirs on purpose.

Combining `compile_inference_path=True` with `precision="bf16"` makes it SLOWER than bf16 eager, not faster (per-block compile's Python/dynamo-guard overhead dominates once bf16 tensor cores already make ops fast) ‚Äî don't bother compiling bf16.

A length landing just under a bucket ceiling pays a real, sometimes-large compute tax on the compiled side alone.
